# Repository Guidelines

## Project Structure & Module Organization
This repo is a reusable GitHub Actions AI PR reviewer ‚Äî no application runtime, all logic lives in workflows plus a small shell/Python toolkit.
- `.github/workflows/` ‚Äî `review.yml` (gate‚Üístatic‚Üícontext‚Üíllm-review‚Üífinalize), `commands.yml` (`/review`, `/plan`, `/oc` router), `ci.yml` (this repo's own static checks).
- `prompts/` ‚Äî LLM playbooks: `review-full.md` and `review-incremental.md` (drafter, mode-specific), `review-verify.md` (skeptic/verifier pass), `review-common.md` (shared protocol appended to both review modes at compose time), `plan.md`. Their `$VAR` contract must match the workflows' `env:` blocks (enforced by `check-contract.py`).
- `scripts/lib/` ‚Äî pure, bats-tested logic the workflows `source` (single source of truth, no copy drift): `reconcile.sh` (baseline/state/thread reconciliation), `sarif.sh` (SARIF‚Üífindings merge), `context.sh` (cross-file impact map), `scope.sh` (`.ai-review.yml` parsing, path matching, size-guard counts), `post.sh` (deterministic review posting ‚Äî verdict derivation, inline budget, anchor validation, GitHub API calls, fallback ladder).
- `scripts/lib/pins.sh` ‚Äî single source of truth for the pinned-tool roster (repo, tag‚Üíversion rule, download-URL template, owning workflow file(s)); sourced by both `check-pins.sh` and `bump-pins.sh` so the list never drifts.
- `scripts/check-pins.sh`, `scripts/check-contract.py` ‚Äî invariant guards run in CI. `scripts/bump-pins.sh <TOOL>` ‚Äî per-tool version+sha256 (or rules-commit) bumper used by `.github/workflows/bump-pins.yml` (weekly + manual; opens one PR per tool, hash recomputed from the asset so the PR is green by construction). That workflow needs a repo secret `BUMP_PINS_TOKEN` (classic PAT, `repo`+`workflow` scopes): the pins live inside `.github/workflows/*.yml` and the default `GITHUB_TOKEN` is forbidden from pushing workflow-file changes. `scripts/release.sh <tag>` ‚Äî release pin bumper. GitHub Action `@sha` pins are auto-updated by `.github/dependabot.yml`.
- `tests/` ‚Äî one `.bats` file per lib + fixtures in `tests/fixtures/`. `templates/` ‚Äî caller workflows for target repos. `rules/` ‚Äî extra OpenGrep rules. `docs/design/` ‚Äî settled design docs (read before reworking the areas they cover). `Makefile` ‚Äî bundles the local check suite.

## Architecture Overview
`review.yml` is a five-job pipeline: **gate** (resolve PR/SHA, skip drafts, fetch `.ai-review.yml` from the **base** branch, config-driven size guard, emit `ignore_patterns`) ‚Üí **static** (opengrep/gitleaks/osv ‚Üí SARIF uploads ‚Üí `findings.json` filtered by config ignores, HIGH severity always kept and annotated) and **context** (ripgrep cross-file impact map honoring ignores ‚Üí `context.md`) in parallel ‚Üí **llm-review** (three steps: (1) **drafter** ‚Äî `opencode run --agent drafter`, no GitHub token, locked down via `OPENCODE_CONFIG_CONTENT`, writes candidate findings JSON; (2) **skeptic** ‚Äî `opencode run --agent skeptic`, fresh session, optional separate model via `verifier_model`/`verifier_variant` inputs, tries to refute each finding, writes verified findings JSON; (3) **deterministic posting** ‚Äî workflow bash sources `scripts/lib/post.sh`, the only step with `github.token`: derives verdict, applies inline budget ‚â§10 blockers/majors, validates comment anchors against the diff, posts ONE review via `gh api`, writes the sticky status comment + `<!-- ai-review:state -->` marker, with a 3-rung fallback ladder for POST failures) ‚Üí **finalize** (`contents:write`; dismisses superseded reviews, resolves threads gated on a fresh, well-formed state). `commands.yml` routes `/review`, `/plan`, `/oc` comments into the same review. State lives in one sticky status comment carrying an embedded `<!-- ai-review:state ‚Ä¶ -->` marker; only the bot-authored marker is trusted, and `reconcile.sh` derives the effective baseline, prior verdict, and open threads from it.

Caller-facing contract: model/provider are `workflow_call` inputs (`model`, `variant`, `api_key_env`) with the secret `LLM_API_KEY`; per-repo behavior comes from `.ai-review.yml` (`ignore:` scopes what the AI reviews, never what the scanners report ‚Äî and it is read from the base branch only, never the PR head, so a PR cannot exclude its own files).

## Build, Test, and Development Commands
Run all checks in one command: `make check` (lint, tests, contract, offline pin check).

Individual commands:
- `actionlint .github/workflows/*.yml` ‚Äî lint workflows (bundles shellcheck on every `run:` block).
- `bats tests/` ‚Äî unit-test the lib scripts (reconcile, sarif, context, scope).
- `python3 scripts/check-contract.py` ‚Äî verify prompt env vars, template permission supersets, and the gate‚Üîlib regex drift-guard.
- `scripts/check-pins.sh` ‚Äî assert all tool version/sha256 pins are in sync and match the live release assets; offline with `CHECK_PINS_OFFLINE=1`.
- `shellcheck scripts/*.sh scripts/lib/*.sh` ‚Äî lint shell.

## Coding Style & Naming Conventions
- Shell: `bash` with `set -euo pipefail`; 2-space indent; `snake_case` functions/vars. Keep `# shellcheck disable=` directives immediately above the offending line (actionlint ignores file-top directives). In workflow `run:` blocks, guard `grep`-based extractions with `|| true` when the key may legitimately be absent ‚Äî an unguarded miss kills the job under `pipefail`.
- Python: 3.12, standard library + `pyyaml`; 4-space indent; type hints; `snake_case`.
- Workflows: pin third-party actions by commit SHA; pin every fetched tool binary by version **and** sha256 (opencode has 3 copies ‚Äî update them together); the OpenGrep ruleset is pinned by commit (`OPENGREP_RULES_REF`). Pass untrusted input via `env:`, never interpolated into `run:`. Never read `.ai-review.yml` from the PR head. When adding/removing a pinned tool, edit `scripts/lib/pins.sh` (the shared roster) so both the checker and the weekly bumper pick it up ‚Äî the per-tool tag quirks (`strip-v` vs `verbatim`, the `apps_v` TAG_SELECT) live there and are unit-tested in `tests/pins.bats`.

## Testing Guidelines
Add a `bats` case for any change to a `scripts/lib/*.sh` function; name tests `area: scenario -> expectation`. Put new fixtures in `tests/fixtures/`. When workflow `run:` blocks compose lib output (grep/cut extraction chains), pin the composition with a smoke test under `set -euo pipefail` ‚Äî the libs' own tests won't catch consumption bugs. Run `make check` before pushing ‚Äî CI runs the same four jobs on every PR (and on pushes to `main`).

## Commit & Pull Request Guidelines
Use Conventional Commits: `feat:`, `fix:`, `docs:`, `ci:` (see `git log`). Keep the subject imperative and scoped to one change. PRs should describe the behavior change, note any pin/version bumps, and pass CI. The project is **alpha**: callers pin an exact tag (e.g. `@v0.2.0`), not a floating major. Release by running `scripts/release.sh <tag>` (bumps every internal `ref:`/`@<tag>` pin and re-verifies ‚Äî README "currently" prose and migration notes are manual), then cutting an annotated tag once `main` is green. After tagging, validate with an e2e acceptance pass in a disposable sandbox repo (caller templates + the tag; seeded-bug PR ‚Üí fix push ‚Üí `/review full` + `/plan` commands ‚Üí size-guard probe) before announcing the release.
